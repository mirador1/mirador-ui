About

Customer Observability UI — full-stack observability, resilience, and management platform

What is this?

A full-stack observability and management platform built with Angular 21 and Spring Boot 4. It demonstrates production-grade patterns for distributed tracing, structured logging, metrics collection, continuous profiling, chaos testing, and resilience — all visible and controllable from the browser.

Architecture Overview

The system is composed of 22 services organized in 6 layers, all orchestrated via two Docker Compose files. Data flows strictly left-to-right: Client → Application → Data Stores → Tools → Collectors → Dashboards.

1. Application Layer

The Customer API is a Spring Boot 4 application running on Java 25 with virtual threads. It exposes a REST API for customer management (CRUD, search, sort, pagination) with two API versions controlled by an X-API-Version header. Authentication uses JWT tokens issued by /auth/login or by Keycloak (OAuth2/OIDC). Every endpoint is instrumented with Micrometer observations, producing 180+ Prometheus metrics automatically.

2. Data Stores

PostgreSQL 17 is the primary database, managed by Spring Data JPA with Flyway migrations. The HikariCP connection pool (10 connections) is fully instrumented. Redis 7 serves three purposes: idempotency key storage (with TTL), the RecentCustomerBuffer (a ring buffer of the last 10 created customers via LPUSH/LTRIM), and rate limit counters (Bucket4j). Apache Kafka (KRaft mode, no ZooKeeper) handles two messaging patterns: async events on customer.created topic, and synchronous request-reply enrichment via customer.request / customer.reply topics (5-second timeout → 504). Ollama runs the llama3.2 LLM locally for AI-generated customer bios via Spring AI, protected by a Resilience4j circuit breaker.

3. Resilience Patterns

The backend implements four resilience patterns visible in the Diagnostic page: Rate limiting (Bucket4j, 100 req/min per IP with 429 response), Circuit breaker (Resilience4j on Ollama /bio — trips after 10 failures in 60s), Retry (on external API calls to JSONPlaceholder), and Idempotency (LRU cache of 10k entries keyed by Idempotency-Key header).

4. Distributed Tracing

Every request generates a distributed trace propagated through HTTP, Kafka, Redis, and JPA. Spring Boot exports traces via OpenTelemetry to the LGTM collector, which forwards them to both Tempo (long-term storage, queried by Grafana) and Jaeger (advanced analysis via OTLP). Zipkin receives traces via its native protocol for its lightweight UI. Kafka observation is enabled (spring.kafka.listener/template.observation-enabled: true), so the traceID propagates through message headers across producer → broker → consumer → reply.

5. Metrics & Profiling

Prometheus scrapes /actuator/prometheus every 15 seconds, collecting 180+ metrics: HTTP request histograms, JVM memory/threads/GC, HikariCP pool stats, Lettuce Redis operations, Kafka producer/consumer timing, Spring Security filter chain, Spring Data repository invocations, and custom application counters. The Angular UI parses this raw Prometheus text format to display 78 configurable metric cards and 55 circular gauges — all built with raw SVG, no charting library. Pyroscope captures continuous CPU, memory allocation, and lock contention flamegraphs via the Java agent (itimer + alloc + lock events).

6. Log Aggregation

Application logs are bridged from Logback to OpenTelemetry via the opentelemetry-logback-appender, then exported through the OTLP collector to Loki inside the LGTM container. The Angular Telemetry tab queries Loki's API directly (via a Nginx CORS proxy on port 3100) with LogQL. Each log line carries the traceID, enabling trace-to-log correlation in Grafana.

7. Admin Tools

Each data store has dedicated admin UIs: pgAdmin and pgweb for PostgreSQL (pgweb provides the REST API used by the SQL Explorer in Settings), RedisInsight and Redis Commander for Redis (Commander adds live command monitoring), Kafka UI and AKHQ for Kafka (AKHQ adds live tail and consumer lag visualization). All tools are pre-configured to auto-connect without login.

8. Docker Infrastructure

The frontend controls Docker containers via a docker-socket-proxy (tecnativa) that exposes a filtered, read-only Docker Engine API over TCP. A Nginx CORS proxy adds cross-origin headers so the Angular app can call it from localhost:4200. The Dashboard's Architecture view shows all 22 services with live UP/DOWN status, Open links, and start/stop controls — all without any custom code in the Spring Boot backend.

Quick Start

# Start everything (backend + frontend)
./run.sh

# Or start individually:
cd ../workspace-modern/customer-service
docker compose up -d                                    # Infrastructure
docker compose -f docker-compose.observability.yml up -d  # Observability
./run.sh app-profiled                                   # Backend with Pyroscope

cd ../customer-observability-ui
npm install && npm start                                # Frontend on :4200
