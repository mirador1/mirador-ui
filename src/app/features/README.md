# `src/app/features/` — Routed pages, grouped by audience

Each subdirectory under the four top-level groups is a **lazy-loaded**
feature, wired in `src/app/app.routes.ts` via
`loadComponent: () => import('./features/<group>/<name>/<name>.component').then(m => m.XComponent)`.
Each feature's JS is fetched only when the user navigates to it.

## Why grouped?

Before this refactor, 15 feature folders sat flat under `features/`, crossing
the global CLAUDE.md subdirectory-hygiene threshold (~12-15 entries, after which
a flat folder stops being scannable). Features are now grouped by primary
**audience** — who the page is for, not what technology it uses — so a new
contributor can reason about the app's structure at a glance:

| Group                       | Audience                                             | Rationale                                                                              |
| --------------------------- | ---------------------------------------------------- | -------------------------------------------------------------------------------------- |
| [`core-ux/`](core-ux/)      | **All users** — app entry points + cross-cutting UX  | The pages every session hits: home, auth, docs hub, backend explorer.                  |
| [`customer/`](customer/)    | **Customer-data users** — CRUD + request playground  | Domain features that operate on the `/customers` entity directly.                      |
| [`obs/`](obs/)              | **Developers / observers** — dev-experience surfaces | Read-only inspectors: audit log, session events, scenarios, build reports, pipelines.  |
| [`infra-ops/`](infra-ops/)  | **Platform operators** — backend ops experiments     | Interactive tools that poke at live infrastructure (DB queries, chaos, OWASP demos).   |

## Conventions

- One folder per feature under one of the four audience groups. Folder name = route
  segment (for the most part — see `maven-site` which serves `/quality/site`).
- Each feature folder ships `<name>.component.ts`, `.html`, `.scss`, `.spec.ts`.
  Larger features may have helpers/subcomponents in the same folder.
- Features must not import from each other. Shared logic goes in `../core/` or
  `../shared/`. Since features now sit one level deeper, relative paths to core/
  and shared/ are `../../../core/` and `../../../shared/` (three `../`, not two).
- Long-form prose (help text longer than ~3 sentences) should live in
  `../../docs/` and be referenced from the component, not inlined. See the
  ongoing doc-extraction effort in `TASKS.md` (when present).
